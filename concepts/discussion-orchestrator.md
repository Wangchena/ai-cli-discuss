# 讨论编排器：群聊模式与共识机制

## 概述

DiscussionOrchestrator 是 AI-CLI-Link 的核心组件，负责协调多个 CLI 实例进行有序讨论并达成共识。

## 群聊模式

### 核心设计

- **完整消息可见性**：每个实例都能看到群内所有消息（包括用户消息和其他实例的回复）
- **顺序轮流发言**：按实例顺序 1→2→1→2 轮流回答，不是并行
- **最多 3 轮**：每个角色最多回复 3 轮，防止无限讨论
- **明确结论要求**：每次回答必须给出明确结论

### 讨论流程

用户提出问题后：

1. 实例 1（第1轮）：基于角色视角分析并给出结论
2. 实例 2（第1轮）：看到实例1的回答，回应并给出结论
3. 实例 1（第2轮）：看到所有历史消息，继续讨论
4. 实例 2（第2轮）：看到所有历史消息，继续讨论
5. ...（最多3轮）
6. 实例 1（总结方）：生成最终总结

### 总结机制

讨论结束后，由第一个回答方（实例 1）负责生成总结：

**达成共识时**：总结共识内容、列出关键决策点、提供下一步建议

**未达成共识时**：总结共同点、列出分歧点、请求用户确认方向

### Prompt 构建

每个实例收到的 prompt 包含：
1. 角色前缀（如 [角色: 架构师]）
2. 完整讨论历史（群聊格式）
3. 当前轮次提示
4. 结论要求

### 共识检测

系统会检测是否达成共识：
- 检测关键词：agree、共识等
- 在第 2 轮之后进行判断
- 如提前达成共识，可提前结束讨论

## 前端展示

### 群聊对话 UI

- **头像系统**：不同角色有不同颜色的圆形头像
- **气泡样式**：类似微信群聊的左右布局
- **实时滚动**：新消息自动滚动到底部
- **轮次标记**：显示当前是第几轮
- **总结高亮**：共识/分歧总结消息居中特殊显示

### 用户随时发言

- 用户可以在讨论进行中发送消息
- 所有实例会看到用户消息并继续讨论
- 讨论完成后可以发送新问题触发重新讨论

## 数据存储

讨论历史保存在 .discussion-history/ 目录，格式为 JSON 文件。
